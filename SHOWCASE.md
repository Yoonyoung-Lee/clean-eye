# Clean Eye in Action

실제 화면을 Clean Eye로 평가한 예시입니다.

> **⚠️ 중요 각주**  
> 아래 화면은 시뮬레이션 예시이며, 실제 제품의 스크린샷이 아닙니다.  
> 이 예시는 Clean Eye가 어떻게 작동하는지 보여주기 위한 것이며, 실제 사용 결과는 평가 대상이 되는 화면마다 다릅니다.

---

## 사례 1: AI 자동 요약 보고서

### 화면 설명

월간 실적 보고서 화면입니다. AI가 자동으로 요약한 텍스트가 상단에 별도 섹션으로 표시됩니다.

**요소:**
- 상단: "AI 요약" 섹션 (자동 생성 배지 포함)
- AI가 생성한 요약 텍스트 (1-2줄)
- 중단: "상세 데이터" 섹션 (펼치기 버튼)
- 하단: "이 요약 사용", "다시 생성", "⋮" 버튼
- 하단: "참고" 정보박스 ("AI 요약은 제공 목적이며, 팀장 승인 필요")

### 전문가 관점

"AI 자동 요약 기능으로 시간을 절약할 수 있고, 사용자가 필요하면 상세 데이터를 볼 수 있으며, '다시 생성' 버튼으로 다른 버전도 시도할 수 있다. 혁신적이고 효율적인 설계다."

### Clean Eye 평가: 일반 직장인 5명 관점

#### 👩🏻 페르소나 1: 32세 여, 마케팅팀 대리, IT 친숙도 높음

**관찰**  
"AI 요약"이라는 섹션이 있고, 요약 텍스트가 있다. 그 아래 버튼이 3개 있다.

**생각**  
"'이 요약 사용'을 누르면 뭐가 되는 거야? 내 이름으로 이 요약이 보고서에 들어가는 건가? 아니면 임시로 보는 거야? '다시 생성'은 또 다른 요약을 만드는 건가?"

**행동**  
일단 "상세 데이터"를 펼쳐서 AI가 요약한 내용이 정확한지 확인해본다.  
그 다음 "이 요약 사용"을 누른다.

**요약**  
AI의 행동이 불명확해서, 일단 데이터로 검증한 후 사용함.

---

#### 👨🏻 페르소나 2: 50세 남, 영업팀 과장, IT 친숙도 낮음

**관찰**  
"AI 요약"이라는 게 있는데... 이게 뭐 하는 건지 모르겠다.

**생각**  
"AI가 만든 요약이라고 했는데, 이게 맞나? 내가 직접 확인해야 하나? '이 요약 사용'을 누르면 즉시 보고서에 반영되는 건가? 아니면 내가 나중에 수정할 수 있나? 혹시 잘못된 정보가 들어가면 누가 책임지는 건데?"

**행동**  
일단 "이 요약 사용"을 누르지 않는다.  
대신 "상세 데이터"를 펼쳐서 직접 요약을 만든다. (AI를 신뢰하지 않음)

**요약**  
AI 기능을 이해하지 못하고, 신뢰도가 낮아서 사용하지 않음.

---

#### 👩🏻 페르소나 3: 38세 여, HR팀 매니저, IT 친숙도 중간

**관찰**  
"AI 요약"이라는 섹션이 눈에 띈다. 아래에 "'AI 요약은 제공 목적이며, 팀장 승인 필요합니다"라는 글이 있다.

**생각**  
"아, 그럼 이 요약이 최종본이 아니고 참고용이군. 그럼 내가 수정해서 팀장한테 올려야 한다는 뜻인가? '이 요약 사용'을 누르면 뭐가 되는 거지? 내 문서에 들어가는 건가?"

**행동**  
"이 요약 사용"을 누르기 전에, 참고 설명을 다시 읽어본다.  
그 다음 "상세 데이터"를 펼쳐본다.  
그 다음 "이 요약 사용"을 누른다.

**요약**  
참고 설명이 도움이 되지만, "이 요약 사용"의 정확한 동작이 불명확함.

---

### 공통으로 걸리는 포인트

**⚠️ 권장 개선 (Top 3)**

1. **"이 요약 사용" 버튼의 동작 명확화**  
   버튼 텍스트를 "이 요약 사용 (팀장 승인 필요)"로 변경하거나,  
   버튼 클릭 후 "다음 단계" 설명 팝업 표시

2. **AI 신뢰도 표시**  
   예: "AI 요약 신뢰도: 87% | 자동 생성됨 · 팀장 검토 필요" 같은 메타데이터 추가

3. **단계별 흐름 명시**  
   "1단계: AI 요약 검토 → 2단계: 팀장 승인 요청 → 3단계: 최종 확정"처럼 프로세스를 명확히 함

### 참고: 페르소나별 세부 차이

- IT 친숙도 높음: 빠르게 이해하지만 여전히 동작에 대한 확인 필요
- IT 친숙도 낮음: AI를 신뢰하지 않아서 사용하지 않으려는 경향

## 사례 2: 회사 휴가 신청 시스템

### 화면 설명

회사 내부 인사시스템의 휴가 신청 대시보드입니다.

**요소:**
- 상단: 연차 잔여, 휴직 잔여, 이월 예정 (3개 카드)
- 중단: 휴가 신청 버튼 (연차, 휴직, 특별휴가, 병가 4개)
- 중단: 휴가 관리 버튼 (신청 내역, 승인 대기, 취소 신청, 연차 조정)
- 하단: 최근 신청 목록 (테이블)

### 전문가 관점

"정보 구조가 깔끔하고, 남은 연차를 한눈에 볼 수 있으며, 각 기능이 버튼으로 분명하게 구분되어 있다. UX 관점에서 좋은 설계다."

### Clean Eye 평가: 일반 직장인 5명 관점

#### 👩🏻 페르소나 1: 28세 여, 인사팀 사원, IT 친숙도 높음

**관찰**  
상단에 숫자들이 있고, 중간에 버튼이 많이 보인다. 연차를 쓰려고 왔는데...

**생각**  
"연차 신청"이라는 버튼이 있네. 이걸 누르면 될 것 같은데, 그 위의 숫자들은 뭐지? 남은 연차? 맞는 것 같기도 하고... 근데 '휴직'도 있고 '특별휴가'도 있는데 뭐가 다른 거지?"

**행동**  
"연차 신청" 버튼을 누름. (맞음)

**요약**  
IT에 익숙한 사람도 처음엔 버튼이 많아서 약간 헷갈림.

---

#### 👨🏻 페르소나 2: 45세 남, 영업팀 과장, IT 친숙도 낮음

**관찰**  
화면이 깔끔한데... 버튼이 많다. 너무 많다.

**생각**  
"나는 연차를 쓰고 싶은데, 어디를 눌러야 하지? '신청 내역'? 아니지, 이건 이미 신청한 거를 보는 거겠지... '연차 신청'인가? 근데 그 위의 3개 박스는 뭐지? 이건 뭔가 중요해 보이는데..."

**행동**  
일단 상단 "연차 잔여" 박스를 눌러본다. (반응 없음)  
그 다음 "신청 내역"을 누른다. (이미 신청한 목록이 보임)  
"아 이건 아니네..."하고 뒤로 간다.  
다시 "연차 신청" 버튼을 눌러본다.

**요약**  
버튼 위계가 불명확해서 여러 번 시행착오를 거쳐야 함.

---

#### 👩🏻 페르소나 3: 52세 여, 행정팀 대리, IT 친숙도 매우 낮음

**관찰**  
글자와 숫자가 많다. 어디서부터 봐야 할지 모르겠다.

**생각**  
"연차가 12일 남았다는 건가? 그럼 내가 휴가를 쓸 수 있다는 뜻인가? 근데 나는 뭘 해야 하지? 이 화면에서 뭘 누르든 휴가가 신청되는 건가?"

**행동**  
화면을 위에서 아래로 천천히 읽다가, "연차 신청" 버튼이 있는 것을 발견하고 눌러본다. (올바른 행동)

**요약**  
정보 밀도가 높아서 처음엔 무엇을 해야 할지 불명확함.

---

### 공통으로 걸리는 포인트

**⚠️ 권장 개선 (Top 3)**

1. **CTA(Call-to-Action) 위계 명확화**  
   "연차 신청"처럼 주요 버튼은 시각적으로 강조해서 첫 번째 행동으로 유도하기

2. **정보 카드에 설명 추가**  
   상단의 "연차 잔여", "휴직 잔여" 같은 숫자에 서브텍스트 ("신청 가능한 일수" 같은) 추가

3. **버튼 그룹핑**  
   "휴가 신청" 그룹과 "휴가 관리" 그룹을 시각적으로 구분하기 (헤더, 색상, 또는 구분선 사용)

### 참고: 페르소나별 세부 차이

- IT 친숙도 높음: 버튼이 많아도 빠르게 찾음 (하지만 아직 약간 헷갈림)
- IT 친숙도 낮음: 여러 번 시행착오 필요

## 결론

Clean Eye의 강점:
- 전문가 관점과 일반인 관점의 갭을 명확히 보여줌
- "시안에는 논리적이지만 사용자는 헷갈린다"는 현상을 포착하기 좋음

한계:
- 합성 페르소나 기반이므로, 실제 사용자 테스트를 완전히 대체할 수 없음
- 정량 데이터 (클릭율, 소요 시간 등)는 제공하지 않음
- 실제 감정, 좌절감, 만족도는 시뮬레이션일 뿐
